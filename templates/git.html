üì¶ What is Git?

Learn the fundamentals of version control and why Git is essential for developers.

ü§î The Problem Git Solves

Imagine working on a document and wanting to save different versions: report_v1.docx, report_v2.docx, report_final.docx, report_final_FINAL.docx...

Git solves this by tracking every change you make, allowing you to:

  • Go back to any previous version
  • See who changed what and when
  • Work with others without overwriting each other's work
  • Experiment safely with new features

üîë Key Concepts

Repository (Repo): A folder that Git tracks. It contains all your files and their complete history.

Commit: A snapshot of your files at a specific point in time. Like a save point in a video game.

Branch: A parallel version of your code. Great for developing features without affecting the main code.

üí° Pro Tip

Think of Git like an "unlimited undo" for your entire project. Every change is recorded, so you can always go back!

üöÄ git init

Create your first Git repository and start tracking your project.

Creating a Repository

To start using Git in a project, you need to initialize a repository. This creates a hidden .git folder that stores all the version history.

Terminal
# Navigate to your project folder cd my-project # Initialize Git git init # Output: Initialized empty Git repository in /my-project/.git/

Checking Status

After initializing, use git status to see what's happening in your repository.

Terminal
git status # Output: # On branch main # No commits yet # Untracked files: # index.html # style.css

üí° Pro Tip

Run git status frequently! It's the best way to understand what Git sees and what your next step should be.

What does 'git init' do?

Creates a new file
Initializes a new Git repository in the current folder
Deletes all Git history

üíæ git add & commit

Learn how to save your changes to the repository.

The Two-Step Save

Git uses a two-step process to save changes:

  1. Stage - Select which changes to include (git add)
  2. Commit - Save those changes with a message (git commit)

git add

The add command moves files to the staging area - like putting items in a shopping cart before checkout.

Terminal
# Add a specific file git add index.html # Add multiple files git add index.html style.css # Add all changed files git add .

git commit

The commit command saves your staged changes with a descriptive message.

Terminal
# Commit with a message git commit -m "Add homepage and styles" # Output: # [main abc1234] Add homepage and styles # 2 files changed, 50 insertions(+)

üí° Good Commit Messages

Write clear messages that explain what changed and why. Future you will thank present you!

Good: "Fix login button not responding on mobile"

Bad: "Fixed stuff" or "asdfasdf"

üîÑ git push & pull

Sync your local changes with a remote repository like GitHub.

Remote Repositories

A remote repository is a version of your project hosted on the internet (like GitHub, GitLab, or Bitbucket). This allows you to:

  • Back up your code in the cloud
  • Collaborate with others
  • Deploy your projects

git push

Upload your local commits to the remote repository.

Terminal
# Push to remote (first time) git push -u origin main # Push after first time git push

git pull

Download changes from the remote repository to your local machine.

Terminal
# Pull latest changes git pull # Output: # Updating abc1234..def5678 # Fast-forward # README.md | 5 +++++

üåø Branches

Work on features independently without affecting the main code.

Why Use Branches?

Branches let you develop features, fix bugs, or experiment in isolation. Think of it like creating a copy of your project to work on, without risking the original.

Common Branch Commands

Terminal
# See all branches git branch # Create a new branch git branch feature-login # Switch to a branch git checkout feature-login # Create AND switch (shortcut) git checkout -b feature-login # Delete a branch git branch -d feature-login

üí° Branch Naming

Use descriptive names: feature-user-auth, bugfix-login-error, hotfix-security-patch

üîó git merge

Combine branches together to integrate your work.

Merging Branches

When your feature is complete, you merge it back into the main branch.

Terminal
# First, switch to the branch you want to merge INTO git checkout main # Then merge the feature branch git merge feature-login # Output: # Merge made by the 'recursive' strategy. # login.html | 25 +++++++++++++++++++++++++ # 1 file changed, 25 insertions(+)

‚ö†Ô∏è Merge Conflicts

Learn how to resolve conflicts when Git can't automatically merge changes.

When Conflicts Happen

Conflicts occur when two branches modify the same line in a file. Git doesn't know which version to keep, so it asks you to decide.

Conflict in file
<<<<<<< HEAD Hello from main branch ======= Hello from feature branch >>>>>>> feature-branch

Resolving Conflicts

  1. Open the conflicted file
  2. Find the conflict markers (<<<<, ====, >>>>)
  3. Choose which code to keep (or combine both)
  4. Remove the conflict markers
  5. Stage and commit the resolved file
After resolving
git add conflicted-file.txt git commit -m "Resolve merge conflict"

üí° Avoiding Conflicts

Pull frequently, communicate with your team, and keep branches short-lived to minimize conflicts.
